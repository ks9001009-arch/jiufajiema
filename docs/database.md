# 数据库设计原则

## 变更管理

- 所有表结构、索引、约束变更必须通过 **migration** 完成。
- **禁止** 直接修改生产数据库。
- Migration 文件存放在 `packages/database`，与业务模块版本对齐。

## 核心数据域（规划）

以下域将在业务阶段逐步建模，当前仅作边界说明：

| 域 | 说明 |
|----|------|
| 用户与团队 | 账号、团队归属、角色关联 |
| 权限 | 角色、资源、策略（逻辑见 [permission.md](./permission.md)） |
| 订单 | 接码订单及状态机 |
| 钱包 | 余额账户与 **流水表**（一切余额变动必经流水） |
| 供应商 | 供应商配置与对接元数据（密钥等存配置层，不入代码） |
| 审计 | audit log、订单操作日志 |

## 余额与流水

- 账户余额不得通过直接 `UPDATE balance` 表达业务语义。
- 每笔变动写入 **钱包流水表**，包含：变动类型、金额、关联业务单号、操作者、时间戳等。
- 余额展示由流水聚合或冗余字段维护，须与流水一致并可对账。

## 订单与日志

- 订单状态使用 **枚举** 定义，禁止在库表或代码中散落中文状态字符串。
- 每次订单状态变化必须写入 **操作日志**（谁、何时、从何状态到何状态、备注）。

## 审计

- 重要操作（登录、权限变更、资金相关、配置变更等）写入 **audit log**，与业务操作日志区分用途时可分表或分类型字段。

## 命名与规范

- 表名、字段名使用 snake_case。
- 主键、外键、索引命名保持一致前缀/后缀约定（实现阶段在 migration 模板中固化）。
- 时间字段统一 UTC 存储，展示层按时区转换。
