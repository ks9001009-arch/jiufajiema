# 权限模型

## 基本原则

- **权限判断必须以后端为准**；前端仅根据后端返回的权限元数据控制 UI 显隐，不得作为安全边界。
- 权限规则 **禁止硬编码** 在页面或零散接口中，应集中配置并可审计。

## 模型方向（规划）

采用 **基于角色的访问控制（RBAC）** 为主，必要时对资源实例做细粒度约束：

```
用户 (User)
  └── 归属团队 (Team)
  └── 持有角色 (Role)
        └── 绑定权限 (Permission)
              └── 资源 + 操作 (resource:action)
```

## 后端职责

| 环节 | 要求 |
|------|------|
| 认证 | 校验身份（会话 / Token，实现阶段选定） |
| 鉴权 | 每个受保护接口在业务逻辑前校验权限 |
| 数据范围 | 按团队/角色过滤可见数据（如仅本团队订单） |
| 审计 | 权限变更、敏感操作写入 audit log |

## 前端职责

- 根据 API 返回的 `permissions` 或等价结构隐藏/禁用按钮与菜单。
- **不得** 在前端实现「是否可扣款」「是否可改价」等核心规则。
- 用户绕过 UI 直接调 API 时，后端必须拒绝未授权请求。

## 与订单、钱包的关系

- 涉及余额、订单状态变更的接口，除 RBAC 外还需校验业务状态机与数据归属。
- 权限枚举与业务状态枚举统一定义于 `packages/shared`。
